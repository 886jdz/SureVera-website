How It All Works -- The Big Picture

Click to expand

SureVera is a mobile app that shows restaurant health inspection grades. Here is how the data flows:

1 Health departments across the US publish inspection data in various formats -- some use open data portals (Socrata, ArcGIS), some publish PDFs, spreadsheets, or CSV files.
2 Scheduled cron jobs on Railway run daily or weekly to download this data from each health department's source. The data is cleaned, standardized into a common format (name, score, grade, date, violations), and stored in a PostgreSQL database.
3 When a user searches (e.g., types a ZIP code), the API does three things simultaneously: (a) queries OpenStreetMap via Overpass to find all restaurants near that location, (b) loads the cached inspection data for that jurisdiction from the database, and (c) fuzzy-matches each restaurant name against the inspection records to find the best match.
4 The matched results are returned to the mobile app, which displays them as a list with letter grades (A through F). The user can tap any restaurant to see detailed inspection history, violation notes, and more.
5 Behind the scenes, the matching algorithm uses Jaccard similarity on tokenized names, with special handling for chain restaurants (Starbucks, McDonald's) that have multiple locations. City enforcement prevents matching a restaurant to another city's inspection record.
6 Data freshness is maintained by the cron schedule -- Socrata/ArcGIS providers refresh daily, custom sources (PDFs, CSVs) refresh weekly. A 30-minute in-memory cache TTL ensures all server replicas serve current data.

Glossary -- Key Terms

Click to expand
OSM (OpenStreetMap)
Free, community-maintained map database. SureVera uses it to find restaurant names and locations.
Overpass API
Query service for OpenStreetMap data. Returns restaurants near a given coordinate.
Nominatim
OpenStreetMap's geocoding service. Converts addresses/ZIP codes to coordinates and vice versa.
Socrata
Open data platform used by many US cities to publish government data. Provides REST APIs.
ArcGIS
Esri's mapping platform. Some health departments publish inspection data via ArcGIS FeatureServer endpoints.
Jaccard Similarity
A text matching algorithm that compares two sets of words. Score = (words in common) / (total unique words). Used to match "McDonald's" to "MCDONALDS #1234 - W SS LOOP".
nominatimCity
The city name from reverse-geocoding a restaurant's GPS coordinates. Used to prevent matching a restaurant to a different city's inspection record.
Jurisdiction Cache
PostgreSQL table storing inspection records per provider. Refreshed by cron jobs, cached in memory with a 30-minute TTL.
Provider
A health department data source. Each provider has a unique ID (e.g., "tx-austin"), a data type (Socrata, ArcGIS, etc.), and configuration for how to fetch and parse its data.
bestMatchFast vs bestMatchInspection
Two matching functions. bestMatchFast is optimized for batch matching (search results, 100+ restaurants). bestMatchInspection is thorough (single restaurant detail view). Both use Jaccard + containsBoost + chain enforcement.
Chain Enforcement
Special matching logic for chain restaurants (Starbucks, McDonald's). Uses city and ZIP code to prevent matching a restaurant to a different location's inspection record.
matchTier
Confidence level of a name match. "confident" (score >= 0.60) = shown to users. "borderline" (0.28-0.49) = hidden from users, available for auditing.
LIVES Format
A standardized CSV format for restaurant inspection data, used by some health departments (Anchorage, Williamson County).
DBPR
Florida Department of Business and Professional Regulation. Publishes statewide inspection data as bulk CSV files.
Railway
Cloud hosting platform where the SureVera API, PostgreSQL database, and cron jobs run.
